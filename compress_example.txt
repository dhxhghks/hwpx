compress_example review

Overview
- This example shows simple GZIP compression and decompression in Kotlin using `GZIPOutputStream` and `GZIPInputStream`.
- The entry point is `compress_example/Compress.kt`.
- It reads `input.txt`, creates `output.gz`, then decompresses that file and writes the restored bytes to `output2.txt`.

Main flow
1. Open `input.txt`.
2. Read the full file into memory with `readBytes()`.
3. Compress the bytes into `output.gz`.
4. Open `output.gz`.
5. Decompress the bytes with `GZIPInputStream`.
6. Write the decompressed bytes into `output2.txt`.

Files involved
- `input.txt`
  Source text file to compress.
- `output.gz`
  GZIP-compressed output generated by the program.
- `output2.txt`
  Decompressed output generated from `output.gz`.

What the example demonstrates well
- Minimal end-to-end GZIP usage in Kotlin.
- Correct pairing of `GZIPOutputStream` for compression and `GZIPInputStream` for decompression.
- Simple proof that decompression restores the original bytes.

Problems and limitations
- File names are hard-coded; command-line arguments are not supported.
- The entire file is loaded into memory for both compression and decompression. That is fine for small files, but not suitable for large inputs.
- `outputString` is created from `outputBytes.toString()`, which does not decode the byte content into text correctly and is never used.
- The code closes streams manually. Kotlin `use {}` blocks would be safer and cleaner.
- There is no error handling for missing files, unreadable files, or invalid gzip content.
- The example does not compare `input.txt` and `output2.txt` to verify equality explicitly.

Practical takeaway
- Use this example as the smallest possible reference for GZIP file compression and decompression in Kotlin.
- For production use, the next improvements should be:
  accept input/output paths as arguments,
  use streaming copies instead of `readBytes()`,
  replace manual close calls with `use {}`,
  and remove or fix the unused `outputString` line.
